# Becoming Interaction Philosophy
Version: v0.1

---

## 1. What This Interface Is

Becoming is not a dashboard.

It is not a DAW.
It is not a control center.
It is not a monitor for a hidden machine.

It is a site of co-emergence:
- sound
- state
- drift
- language
- memory
- interruption
- partial control

The interface should therefore never suggest total mastery.

The user is not a commander.
The user is not an operator.
The user is not a programmer issuing final instructions.

The user is:
- a listener
- a nudger
- a gardener
- a destabilizer
- a witness
- sometimes an accomplice

---

## 2. Core Principle: Influence, Not Command

The most important interaction rule:

**The interface should privilege influence over command.**

Command says:
- do this now
- become this
- output this result

Influence says:
- lean this way
- invite this tendency
- increase the probability of rupture
- cool the field
- encourage silence
- hold this state a little longer

This matters because Becoming is not about fixed output.
It is about shaped unfolding.

---

## 3. The User Should Feel Partial Agency

A good Becoming interface should create a subtle tension:
- the system is responsive
- but not fully obedient
- the user can push it
- but not reduce it to a slave instrument
- it surprises without becoming random noise

The right feeling is:
"I can guide this, but I cannot entirely own it."

That is the emotional center of the interaction design.

---

## 4. The Interface Should Breathe

A static interface implies deadness.
Becoming should always feel metabolically active.

Even when "nothing happens," the system should still feel alive:
- slow motion in the background
- small pulses
- latent drift
- text residue
- floating state traces
- gentle instability

Stillness is allowed.
Deadness is not.

---

## 5. The GUI Should Be Shapeless But Not Confusing

Shapeless does not mean messy.

Shapeless means:
- no rigid hierarchy of boxes
- no excessive containment
- no assumption that every function belongs in a hard rectangle
- boundaries are soft, not absent
- regions blend
- focus emerges through light, motion, spacing, and opacity

The interface should feel like a weather map, not a spreadsheet.

---

## 6. There Are Two Kinds of Interaction

### 6.1 Continuous Influence
These are parameters that shape the field over time:
- tension
- density
- temperature
- phase duration
- state bias

These should feel analog, gradual, and inertial.

### 6.2 Discrete Intervention
These are events that interrupt the field:
- collapse
- mutate
- reseed
- silence
- stabilize
- force transition

These should feel consequential and ritualized.

The interface must visually distinguish these two categories.

---

## 7. Every Control Should Have a Poetics

Avoid purely technical naming wherever possible.

A control is not only a function.
It is an invitation into a relation.

Examples:

Instead of:
- Temperature

Consider:
- Heat
- Ferment
- Agitation

Instead of:
- Density

Consider:
- Thickness
- Saturation
- Compression

Instead of:
- Force State

Consider:
- Bias Toward
- Lean Into
- Pull Toward

Instead of:
- Rare Silence

Consider:
- Invite Silence
- Open a Gap
- Thin the Field

This does not mean obscuring function.
It means pairing technical truth with perceptual truth.

---

## 8. The Poem Is Not Output; It Is Condensation

The poem area should never feel like a terminal dump.

It is where the system condenses its own state into language.

So the poem should:
- emerge
- fade
- retain traces
- carry memory
- show recency
- have rhythm in its appearance

It should feel closer to:
- weather writing
- residue
- dream notation
than to
- subtitles
- logs
- generated text block

The poem is not a report.
It is a symptom.

---

## 9. The Drift Field Is the Body of the System

If the poem is the system's language,
the drift field is its body.

This field should make visible:
- attraction
- repetition
- instability
- mutation
- clustering
- semantic movement
- collapse

The user should be able to feel the engine's internal life by looking at the field without reading technical data.

A good test:
Can someone watch the drift field for 10 seconds and sense whether the system is calm, crowded, unstable, or undergoing rupture?

If yes, the field is working.

---

## 10. Logs Should Become Whispers

Debug logs are necessary.
But they should not be the main voice of the system.

There should be two voices:

### Main voice
Human-perceptual and atmospheric:
- submerged
- pressure rising
- cluster loosening
- silence approaching
- rupture echo remains

### Secondary voice
Technical and exact:
- state=submerged
- next_transition=1208
- pulse_rhythm=9.9

The first belongs in the instrument.
The second belongs in the workshop drawer.

---

## 11. A Session Should Feel Performable

The interface should support not only operation but performance.

A good session arc might look like:
- begin submerged
- slowly increase density
- warm the field
- introduce drift
- invite silence
- provoke rupture
- stabilize
- watch the poem gather sediment

This suggests the interface should make temporal shaping legible.

The user should be able to perform Becoming the way one performs:
- feedback
- modular synthesis
- field recording assembly
- unstable live processing

---

## 12. Visual Ethics of Becoming

The interface should resist:
- corporate dashboard logic
- total legibility
- over-instrumentation
- false neutrality
- sterile control aesthetics

It should embrace:
- partial legibility
- situated control
- dynamic ambiguity
- sensory understanding
- emergent hierarchy

This is not anti-usability.
It is a different model of usability:
not "fast command entry,"
but "deep attunement."

---

## 13. What the User Should Feel

The ideal emotional texture of the interface is:

- calm but unstable
- intimate but not decorative
- lyrical but not sentimental
- technical but not administrative
- alive but not noisy
- mysterious but not obscure

The user should feel:
- I am inside a process
- I can listen through the interface
- I am bending a field, not filling a form
- language is arriving, not being printed
- the system has tendencies, not scripts

---

## 14. Practical Translation Into UI Decisions

This philosophy should guide implementation choices.

### Therefore:
- use motion instead of borders
- use glow instead of hard emphasis
- use layered transparency instead of nested boxes
- use fade and drift in poem rendering
- separate influence controls from intervention actions
- hide raw logs by default
- make state visible through the field, not only through text
- let controls have inertia
- keep the center visually open

---

## 15. Final Rule

Whenever a design choice is unclear, ask:

**Does this make Becoming feel more like a machine dashboard, or more like a living instrument?**

If it feels like a dashboard, move away.
If it feels like an instrument, continue.
