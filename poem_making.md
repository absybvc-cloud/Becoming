# Poem Making — Live Poetic Visualization

## 1. Problem

Becoming produces a layered ecology of sounds, but there is no human-readable
narration of what is happening _right now_. The listener hears but cannot
"read" the sonic moment. We want to turn the live state — the tags, roles,
clusters, and drift phase of every simultaneously playing sound — into an
evolving line of poetry, generated by a local LLM.

## 2. Core Idea

Every N seconds (configurable, default 15s), the system:

1. **Harvests** the current active layers from the conductor: for each
   playing fragment, collect its tags, role, cluster, age, and gain.
2. **Distills** these into a word-cloud: a flat set of the most evocative
   words currently sounding, plus the engine state name, drift phase, and
   tension/density values as mood modifiers.
3. **Prompts** a local Ollama model (same infrastructure as auto-tag) with
   the word-cloud and a short system instruction asking for one poetic line.
4. **Appends** the returned line to a scrolling text widget in the GUI,
   building an ongoing poem that narrates the piece in real time.

## 3. Word Harvest

From each active layer, extract:

| Field | Source |
|---|---|
| tags | `SoundFragment.tags` (list of curator + model tags) |
| role | `ActiveLayer.role.value` (ground / texture / event / pulse) |
| cluster | looked up via `library.get_cluster(fid)` |
| age | `ActiveLayer.age` in seconds |
| gain | `ActiveLayer.gain` |

Additional context:
- **state** — `state_machine.current` (submerged, drifting, surfacing, rupture, suspended)
- **drift phase** — `drift_engine.snapshot()["phase"]` (stabilize, drift, saturation, collapse, reconfiguration)
- **tension / density** — `world.get().tension`, `world.get().density`

### Tag Filtering

Strip generic/mechanical tags (`drift_material`, `background_layer`,
`found_sound`, `noise`, `competition`, etc.) to keep only evocative words.

## 4. Prompt Design

### Rhyme Scheme

The poem follows a **couplet** pattern (AABB): lines are generated in
pairs, where each pair rhymes. On an **odd** line (1st of the pair) the
LLM writes freely. On the **even** line (2nd of the pair) the LLM must
end with a word that rhymes with the last word of the previous line. The
GUI tracks the couplet index to alternate between free and rhyming modes.

### Beat (Ascending / Descending)

The directional beat still alternates per line: odd lines RISE, even
lines FALL. This gives the couplets an inhale/exhale shape.

### Prompt Template

```
You are a poet channelling an ever-shifting sound ecology called "Becoming".
These sounds are alive right now:

{word_cloud}

Mood: {state} | Phase: {phase} | tension={tension} | density={density}

{direction}
{rhyme_instruction}

Write ONE vivid poetic line — bold imagery, unexpected metaphor, sensory
language. No quotes, no explanation, just the raw line.
```

When it is a **rhyming** line, `{rhyme_instruction}` becomes:

```
Your line MUST end with a word that rhymes with "{rhyme_word}".
Previous line to rhyme with: "{previous_line}"
```

When it is a **free** line, `{rhyme_instruction}` is empty (or simply
"Write freely.").

## 5. LLM Integration

- Use the same Ollama endpoint as auto-tag: `http://127.0.0.1:11434/api/generate`
- Model: configurable, default `qwen3-coder:30b` (or whichever is set in GUI)
- Temperature: `1.0` (high creativity — poems should be bold)
- `num_predict`: `80` (enough room for a rich line)
- Timeout: `30s` (fast or skip)
- If the call fails or times out, skip that tick silently.

## 6. GUI Integration

### New card: "Poem Making" on the Engine tab

- Scrolling `tk.Text` widget (read-only, dark background, poetic font)
- **Start Poem** / **Stop Poem** toggle button
- **Interval** slider or entry (5–60 seconds, default 15)
- Each new line appended with a subtle timestamp prefix (`HH:MM`)
- Auto-scroll to bottom
- Max 200 lines kept (oldest trimmed)

### Architecture

- A daemon thread (`_poem_thread`) runs a loop:
  - Sleep for `interval` seconds
  - If engine is running, send `poem\n` to engine stdin
  - Engine responds with a `[poem-data] {...}` JSON line on stdout
  - GUI reader picks it up, calls Ollama, appends the result
- Alternatively (simpler): the poem thread directly calls Ollama from
  the GUI process, using the engine stdout lines to harvest active layer
  info from `[status]` lines already being printed.

**Chosen approach**: Add a `poem\n` stdin command to the engine that outputs
a `[poem-words] {...}` JSON blob with the word cloud. The GUI poem thread
sends this command periodically, parses the response from the log stream,
calls Ollama, and displays the line.

## 7. Engine-side: `poem` command

In `engine.py` `input_listener`, handle `poem`:

```python
elif cmd == "poem":
    layers = conductor.active_layers
    words = set()
    for fid, layer in layers.items():
        words.update(layer.fragment.tags)
        words.add(layer.role.value)
        cluster = library.get_cluster(fid)
        if cluster:
            words.add(cluster)
    words -= FILTER_TAGS
    state = state_machine.current
    phase = drift_engine.snapshot().get("phase", "drift")
    w = world.get()
    blob = {
        "words": sorted(words),
        "state": state,
        "phase": phase,
        "tension": round(w.tension, 2),
        "density": round(w.density, 2),
    }
    print(f"[poem-words] {json.dumps(blob)}")
```

## 8. Module: `poem_maker.py`

Standalone module with:

- `harvest_words(blob: dict) -> str` — format word cloud for prompt
- `build_poem_prompt(words_str, state, phase, tension, density) -> str`
- `generate_line(prompt, model, ollama_base) -> str | None` — call Ollama
- `FILTER_TAGS` — set of tags to exclude from the word cloud

## 9. Output Example

```
14:32 ▲ the copper bell unravels into rain
14:32 ▼ a thousand wires hum beneath the plain
14:33 ▲ bright fractures climb the throat of burning air
14:33 ▼ the static folds into a whispered prayer
14:34 ▲ volcanic glass dissolves along the shore
14:34 ▼ each silence learns to be a little more
```

Lines alternate ▲ (ascending/RISE) and ▼ (descending/FALL). Each pair
of consecutive lines rhymes (AABB couplet pattern).
